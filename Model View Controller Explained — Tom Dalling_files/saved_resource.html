                                                                                                                                Tom Dalling Mod Space Cowboy 3 years ago

                                                                                                                                So you're talking about persistence, and persistence commonly happens in either the model or the controller. People will disagree with me about this, but it should really go in the controller. You should have one or more external services (for example, a database) that the controller can pull model data from (querying) and push model data to (persisting).

                                                                                                                                I think this becomes apparent when you start working with multiple data stores. Think about what happens when the model is a combination of data from different places. Also think about what happens when you need to persist the same model to multiple places, such as a database and an in-memory cache.

                                                                                                                                                          As it stands today, If you use struts for eg, there is only deps in code between C and M and between V and M. Well the controller name cld be mentioned in the form in old school struts or via annotations in the C in new-school struts config xml.. But that ain't no hard dep. So Views are reusable and if written with HTML and placeholders(templating), views will be reusable across languages too. Unit Testing of a View is hard with MVC because of the fact that it is outside the realm of the lang & technology used for coding the controller and model. Both controller and model code methods are unit testable. Views can be exercised using selenium for eg.

                                                                                                                                                          Spring when used for webapps uses a viewmodel but it is more like a dto. Hence can't call that a MVVM setup. The C there is dep on the viewmodel and the M. It is still MVC. For swing/javafx it is generally the pattern.. Here the Presenter is dependent on View components like say a JTree orthe TreeModel. You cld use the listener, observable pattern to break hard code deps between presenter and view.
